# TaskDisplayの動作

- `xModeQueue` から受け取った値によって、**3パターン**に分岐

## 受信時の3パターン

```text
受信した modeMsg
        │
        ├─ MODE_EYE ──────────────────────────→ 即座に目に戻る
        │
        ├─ modeMsg == 現在の mode（再押し）──→ 即座に目に戻る
        │
        └─ 新しいモード（例: MODE_CLOCK）──→ 切替 + 5秒タイマーをセット
```

### パターン1：`modeMsg == MODE_EYE`

- 明示的に「目に戻れ」という指示が来た場合
  - `mode = MODE_EYE` に即切替

### パターン2：`modeMsg == mode`（再押し検出）

- **すでに時計を表示している最中にボタンをもう一度押した**場合
  - 「同じモードが来た＝キャンセル」と解釈し `mode = MODE_EYE` に戻す

### パターン3：新しいモードへの切替

```cpp
mode    = modeMsg;              // 例: MODE_CLOCK に切替
showEnd = now + MODE_SHOW_MS;   // now + 5000ms = 5秒後を期限として記録
```

- `showEnd` に「5秒後のミリ秒時刻」を仕込んでおくのがポイント

## キュー受信後の自動タイムアウト

- キュー処理の直後に毎ループこのチェックが走る

```cpp
if (mode != MODE_EYE && millis() >= showEnd) {
    mode = MODE_EYE;  // 5秒経過 → 命令なしで自動復帰
}
```

- `delay()` を使わず `millis()` で時刻比較するので、**タイムアウト中も描画ループは止まらず15fps を維持**可能

## なぜ待機時間を 0 にしているか

```cpp
xQueueReceive(xModeQueue, &modeMsg, 0)  // 第3引数 = 0
```

- 待機時間を0にすることで「キューが空でもブロックせず即リターン」する
- もしここで `portMAX_DELAY`（無限待機）にすると、ボタンが押されるまでループが止まり、目のアニメーションが完全にフリーズしてしまうので注意
